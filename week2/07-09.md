> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

    - /mypage/index.html

  - 공통 라우터 클래스
  - html 입력값 검증
### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 동적 html요소에 들어갈 html 부품을 컴포넌트로 분리해서 html파일 관리.
- 그렇게 판단한 이유: 컴포넌트로 분리해서 관리하는게 여러 코드에서 공통으로 사용하기 편하기 때문이고, 에디터의 Linting과 자동완성을 사용할 수 있기 떄문이다.
- 고려했지만 선택하지 않은 대안: 각 페이지 핸들러에서 html 문자열 관리하기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 컴포넌트 수정할 때 각 핸들러 가서 수정하지 않고 컴포넌트 html 파일을 수정하면 된다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: enum으로 컴포넌트 파일 이름 및 경로관리.
- 그 접근이 도움이 된 이유: html 파일 불러와서 조립할 때 경로 실수 가능성을 없애고, 추가도 쉽게 할 수 있다.
- 다음에도 반복하고 싶은 점: 프로그램 실행 중 변화가 없는 값에 대해서는 enum 클래스를 적극 활용하자.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 동적 html 권한 확인을 각 핸들러에서 처리함.
- 그렇게 된 원인: 기능 구현하는 걸 우선 확인하고 나중에 공통되는 부분이 생기면 바꾸자 생각했음. 공통되는 부분이 당연하게 발생했음.
- 다음에는 다르게 해볼 구체적인 행동: 리팩토링 할 일 없게 처음부터 공통된 부분 따로 개발하기.
