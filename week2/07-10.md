# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 리팩토링: index 파일 주소를 HtmlResource enum으로 관리
  - 캐시 구현: 컴포넌트와 index html 파일을 Cache 클래스에 static map에서 관리
    - 목적: 디스크 I/O 감소
    - Lazy Loading 대신 Eager Loading 채택. 서버 최초 구동시 캐시 로딩
  - 학습:
    - 하드 코딩을 피하는법 (상대경로, 환경변수, Enum)
    - ENUM 클래스: 일반 클래스 상속은 안되고, 인터페이스 구현은 가능
    - 인터페이스: default 메서드. this는 못씀. getter 추상메서드를 default안에 넣어서 구현 가능
    - 캐싱의 목적: 디스크 I/O 오버헤드 감소
    - 정적 파일 캐싱: 서버가 아닌 브라우저나 CDN에서 -> 네트워크 I/O 오버헤드 감소
    - Map.computeIfAbsent
    - Checked Exception은 람다식 밖으로 전파되지 않는다.
    - BufferedReader + 스트림 조인 vs. readAllbytes + String 생성
      - 전자: 줄바꿈 처리 제어 가능, 메모리 효율성 / 느리고 한줄마다 GC, 원형 파괴 위험
      - 후자: 속도, 원본 보존 / 대용량 파일 OutOfMemoryError 가능성.
    - public final을 써도 될 때와 안될 때.
      - String이나 원시타입이면 괜찮음. 가변 객체면 안 됨.
      - 실무 표준은 전부 private -> 일관성 있고 리팩토링도 편함. 롬복쓰면 getter 자동
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 정적 파일 캐싱

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: html 파일 주소 enum으로 관리
- 그렇게 판단한 이유: 타입 안전성
- 고려했지만 선택하지 않은 대안: 핸들러 안에서 문자열로 관리
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 파일 주소가 바뀔 때 enum 클래스만 수정하면 됨. 오타 걱정도 줄어듦

### 3. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 스쿼드세션으로 나와 다른 코드에 대한 걸 많이 물어봤다.
- 그 접근이 도움이 된 이유: 코드에서 분명하게 잘못됐지만 찾지 못한 부분을 찾을 수 있었고, 당연하게 넘어간 부분에 대한 지식도 습득할 수 있었으며, 유용한 단축키도 알았다.
- 다음에도 반복하고 싶은 점: 그룹세션, 스쿼드 세션 아니어도 대화 많이 하기.
