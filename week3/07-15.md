- 학습 (오늘 사용한 함수 간단 정리):
  - `Arrays.equals(byte[] a, int aFromIndex, int aToIndex, byte[] b, int bFromIndex, int bToIndex)`: 두 배열의 지정한 구간끼리만 비교. 배열을 잘라서 복사하지 않고도 body 안의 특정 위치에 boundary 바이트 패턴이 있는지 확인할 때 씀.
  - `Arrays.copyOfRange(byte[] original, int from, int to)`: 배열의 `[from, to)` 구간만 새 배열로 복사해서 반환. multipart body를 boundary 기준으로 part 하나씩 잘라낼 때 씀.
  - `Files.deleteIfExists(Path path)`: 파일이 있으면 지우고 없으면 조용히 넘어감(예외 안 던짐). 이미 지워졌을 수도 있는 이미지 파일을 안전하게 정리할 때 씀.
  - `URL.createObjectURL(file)` (JS): 로컬에서 고른 `File` 객체를 가리키는 임시 blob URL 생성. 서버에 아무것도 안 보내고 `<img src="">`에 바로 꽂아서 미리보기 구현할 때 씀.
  - `element.click()` + `addEventListener("change", ...)` (JS): 숨겨둔 `<input type="file">`을 스크립트로 강제 클릭해서 열고, 사용자가 실제 파일을 고르면 브라우저가 채워주는 `change` 이벤트로 결과를 읽는 방식.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 이미지를 DB에 BLOB으로 저장하지 않고, 파일시스템(`./data/images`)에 저장한 뒤 DB에는 파일명만 저장하기로 함.
- 그렇게 판단한 이유: H2는 단일 파일(`.mv.db`) 구조라 이미지 바이트까지 쌓이면 DB 파일이 급격히 커지고, 서빙할 때마다 DB round-trip이 필요해서 정적 파일 서빙보다 비효율적임. 대용량 바이너리는 파일시스템/오브젝트 스토리지에 두고 DB엔 참조만 두는 게 실무에서도 흔한 패턴.
- 고려했지만 선택하지 않은 대안: DB에 BLOB 컬럼으로 직접 저장 (트랜잭션 원자성은 얻지만 DB 비대화와 서빙 성능 저하가 더 큰 단점).
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: DB는 가볍게 유지되고, 나중에 S3 같은 오브젝트 스토리지로 저장소를 교체해도 `ImageStorage` 내부만 바꾸면 되는 구조가 됨. 대신 파일-DB 정합성을 직접 관리해야 하는 부담이 생겨서 별도 작업으로 남겨둠.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `multipart/form-data` 바디를 직접 파싱하는 과정에서, boundary로 파트를 나눌 때 헤더/데이터 경계, 앞뒤 `\r\n` 처리, `Content-Disposition`의 `name`/`filename` 값이 따옴표로 감싸져 있는 부분을 놓치기 쉬웠음.
- 처음 추측한 원인: boundary 매칭 위치만 찾으면 그 사이를 그냥 잘라서 쓰면 될 거라 생각함.
- 원인을 확인하기 위해 시도한 방법: boundary 매칭 인덱스를 리스트로 모아서 인접한 쌍(pair)마다 part 하나씩 잘라내는 구조를 직접 바이트 단위로 그려보며 검증.
- 실제 원인: (1) part 데이터 끝의 `\r\n`은 데이터가 아니라 다음 구분자의 일부라 별도로 잘라내야 했고, (2) `name`/`filename` 값을 꺼내면 따옴표가 그대로 남는데 이를 벗기기 전에 유효성 검사를 해서 `name=""` 같은 빈 값을 못 걸러냄.
- 해결 방법: boundary 매칭 쌍 사이에서 앞뒤로 정확히 필요한 바이트만 슬라이싱하고, `stripQuotes` 헬퍼로 따옴표를 먼저 벗긴 뒤 유효성 검사하는 순서로 수정. 종료 boundary(`--`) 검증과 매칭 리스트가 빈 경우에 대한 방어도 추가.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 텍스트 기반 프로토콜을 바이트 레벨로 파싱할 땐 "구분자가 데이터에 포함될 수 있는지"와 "값을 감싸는 문자(따옴표 등)가 있는지"를 스펙에서 먼저 확인하고 시작하기.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 코드를 짜기 전에 전체 단계(파싱 → 저장 → DB → 서빙 → 통합 → 정합성)를 먼저 계획하고, 확장자 화이트리스트/크기 제한/반환 구조 같은 애매한 값은 미리 정하고 각 단계마다 피드백받으며 진행함.
- 그 접근이 도움이 된 이유: 중간에 설계를 뒤엎는 일이 없었고, 파서/저장/서빙 계층이 서로 독립적으로 재사용 가능한 구조로 나옴.
- 다음에도 반복하고 싶은 점: 파서처럼 핵심 로직을 다 짜고 나서 실제 HTTP 요청 바이트를 흉내낸 단위 테스트로 바로 검증한 뒤에 그 위에 다음 기능(저장 계층)을 쌓은 것. 검증 없이 계속 쌓았으면 버그 위치를 찾기 훨씬 어려웠을 것.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: `ImageStorage`의 이미지 확장자 화이트리스트(jpg/jpeg/png/gif/webp)를 정할 때, 기존 `HttpResponse.mimeTypes`엔 jpg/png만 있고 gif/webp가 없다는 걸 그 시점에 같이 확인하지 않음. 나중에 서빙 핸들러를 만들 때서야 `Content-Type: null`이 나가는 버그로 발견함.
- 그렇게 된 원인: 허용 목록을 새로 추가할 때 그 값이 다른 곳의 기존 매핑 테이블과 짝을 맞춰야 하는지 확인하지 않고 해당 계층만 보고 결정함.
- 다음에는 다르게 해볼 구체적인 행동: 허용 목록(화이트리스트, enum 값 등)을 추가하거나 넓힐 때는, 그 값이 나중에 반대 방향으로(이번엔 확장자 → MIME 타입) 조회되는 곳이 있는지 grep으로 먼저 확인하고 넘어가기.
