# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - java.util.concurrent.ExecutorService(스레드 풀)를 웹 서버 메인 루프에 도입하여, 클라이언트 요청이 폭주해도 지정된 개수(250개) 내에서 안전하게 재사용되도록 고성능 멀티스레드 환경 구축 및 원리 학습.
  - getClass().getResourceAsStream()을 사용하는 클래스패스(Classpath) 기반 리소스 스트림 방식의 개념을 정립하고, try-with-resources 문법을 적용하여 자원 누수(Memory Leak)가 없는 안전한 파일 I/O 구현.
  - 인텔리제이 내에서 프라이빗 레포지토리 권한 이슈 및 Git 업스트림(Upstream) 개념을 파악하여 특정 브랜치로 커밋/푸시 대상을 고정하는 문제 해결.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 요청한 정적 파일이 서버 리소스에 존재하지 않을 경우(404 Not Found), 바디 내용만 교체하는 대신 규격에 맞는 HTTP/1.1 404 Not Found 프로토콜 헤더를 분리하여 응답하는 예외 처리 구조 고도화.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 로컬 개발 단계에서 java.io.File을 사용하는 프로젝트 상대 경로 대신, 처음부터 빌드 및 배포 환경(JAR)까지 에러 없이 완벽하게 커버할 수 있는 클래스패스 리소스 스트림(getClass().getResourceAsStream()) 방식을 프로젝트의 최종 표준 코드로 채택한 것.
- 그렇게 판단한 이유: File 객체 경로는 로컬 IDE에서는 잘 돌아가도 나중에 하나의 실행 파일(JAR)로 압축해서 배포하면 리소스 폴더 구조가 패키지 내부로 꽁꽁 숨어버려 무조건 크래시(Crash)가 나기 때문. 어차피 배포 단계에서 무조건 고쳐야 할 코드라면 처음부터 안전한 표준 방식을 확립해 두는 것이 옳다고 생각함.
- 고려했지만 선택하지 않은 대안: 현재 개발 생산성(HTML/CSS를 수정하고 서버 재시작 없이 브라우저 새로고침만으로 화면을 바로 확인하는 편리함)을 위해 new File("src/main/resources/static/...") 방식을 그대로 유지하는 대안.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 비록 개발 과정에서 정적 파일을 수정할 때마다 서버를 껐다 켜야 하는 약간의 번거로움(피드백 루프)은 생겼지만, 환경(로컬 PC, 클라우드, 리눅스 서버 등)에 구애받지 않고 어디서나 독립적이고 일관되게 100% 작동하는 견고하고 안전한 코드 품질을 확보함.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 스레드 풀을 분명 적용했음에도 새로고침을 하거나 반복 요청을 보낼 때 로그에 찍히는 스레드 번호(pool-1-thread-30 등)가 고정되지 않고 지속적으로 증가하는 현상 발생.
- 처음 추측한 원인: ExecutorService가 작동하지 않고 내부적으로 스레드가 해제되지 않은 채 계속 새로 생성되고 있는 버그라고 추측함.
- 원인을 확인하기 위해 시도한 방법: 브라우저 새로고침을 짧은 시간에 연속으로 타격하며 로그 창에 찍히는 스레드 네이밍 규칙과 증가 폭을 관찰함.
- 실제 원인:
  - 브라우저가 유저의 눈에 보이지 않게 비동기식으로 여러 정적 자원(HTML, CSS, Favicon 등)을 동시다발적으로 요청했기 때문이었고
  - 스레드 풀은 들어오는 요청들을 라운드 로빈 방식으로 유휴 스레드에 골고루 분배하기 때문에 번호가 순차적으로 올라갔던 것임.
- 해결 방법: 스레드 풀이 최대 크기(250개) 내에서 정상 작동하고 있음을 디버깅 로그로 확인하고, 요청을 연속 연타했을 때 기존에 생성되었던 앞 번호 스레드가 정상적으로 재사용(Reuse)되는 메커니즘을 최종 확인하여 정상 동작임을 검증함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 네트워크 탭을 열어 클라이언트(브라우저)가 실질적으로 서버에 보낸 총 Request의 개수가 몇 개인지 먼저 카운트해 볼 것.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 스레드 풀을 구현하라는 지시를 받았을 때 코드에 무작정 submit()만 붙이는 데 그치지 않고, 왜 스레드 번호가 증가하는지 의문을 품고 그 기저에 깔린 브라우저의 동작 원리와 스레드 풀 배정 알고리즘까지 파고들어 확인한 점.
- 그 접근이 도움이 된 이유: 단순히 "돌아가는 코드"를 카피하는 것을 넘어 멀티스레드 환경에서 자원이 어떻게 관리되는지 본질적인 동작 흐름을 이해하게 됨.
- 다음에도 반복하고 싶은 점: 코드가 예상과 다르게 움직일 때 당황하지 않고 로그의 디테일한 힌트(스레드명 등)를 기반으로 역추적하는 집요함 유지하기.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: executorService.submit(new Thread(new RequestHandler(connection))) 형태로 코드를 작성하여 Thread 객체와 스레드 풀의 역할을 혼동했던 점.
- 그렇게 된 원인: 스레드 풀은 '일감(Runnable)'을 관리하는 주체라는 본질을 망각하고, 기존에 사용하던 new Thread().start() 구조를 기계적으로 대입하려다가 생긴 인터페이스 오용
- 다음에는 다르게 해볼 구체적인 행동: 새로운 라이브러리나 패키지(java.util.concurrent)를 도입할 때는 무작정 객체를 욱여넣기 전에 해당 클래스가 요구하는 인자의 역할(Task인가 Worker인가)을 공식 문서나 개념 정의를 통해 명확히 선언하고 출발하기.
