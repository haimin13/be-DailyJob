### 2. 오늘 가장 의미있었던 기술적 판단
- 판단: HttpResponse 클래스를 만들어서 관리하기
- 이유: handler가 결과 body뿐 아니라 상태 코드와 같은 여러 정보를 반환해야 할 필요성을 느꼈다.
- 대안: handler가 body만 반환하도록 유지하고, 메소드 외부에서 body를 읽고 상태코드 등 헤더를 판단해서 response header에 포함하는 방식
- 장점: 단일 책임 원칙을 지키면서, 메소드 안에서 지저분한 헤더 요구사항을 다 작성하도록 해서 main requestHandler 코드의 가독성을 유지하고, handler를 추가할 때도, 동일한 원칙으로 작성 가능

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 나만의 WAS(웹 서버)를 구현하는 과정에서, 클라이언트로 데이터를 전송할 때 콘솔에 Broken Pipe (java.io.IOException) 에러가 지속적으로 발생하거나 브라우저가 응답을 정상적으로 처리하지 못하고 연결을 끊어버림.
- 처음 추측한 원인: 응답 데이터의 길이(Content-Length) 계산이 잘못되었거나, 멀티스레드 환경에서 임의로 상태를 공유하여 동시성 문제가 발생한 것으로 추측함.
- 원인을 확인하기 위해 시도한 방법: 헤더를 구성하는 responseHeader 메서드의 로직을 뜯어보고, 상태코드(code)와 상태 메시지(statusMsg), 그리고 body.length가 결합하여 DataOutputStream으로 어떻게 전송되는지 문자열 조립 과정을 디테일하게 추적함.
- 실제 원인: 두 가지 복합적인 원인이었음. 첫째, HTTP 응답의 첫 줄(Status Line)을 조립할 때 code와 statusMsg 사이에 필수 공백( )을 누락하여 브라우저가 400Bad Request 같은 잘못된 스펙으로 인식해 연결을 끊어버림. 둘째, 바디가 없는 응답일 때 res.getBody().length를 호출하여 NullPointerException이 발생해 스트림이 비정상 종료됨.
- 해결 방법: 상태 줄 조립 시 명시적으로 공백 문자열(" ")을 추가하여 HTTP 스펙을 준수하도록 수정함. 또한, 바디 배열이 null일 때를 대비해 삼항 연산자를 활용한 방어 코드((body != null) ? body.length : 0)를 추가하여 안전하게 길이를 구하도록 리팩토링함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 네트워크 통신 에러나 파이프가 깨지는 현상이 발생하면, 내가 조립한 HTTP 메시지의 텍스트 포맷(공백, 줄바꿈 \r\n)이 표준 규격에 완벽히 맞는지, 그리고 스트림을 다루기 전 Null이 발생할 여지가 없는지 최우선으로 검증할 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: HttpResponse 클래스 작성, static 필드, 메소드와 멀티스레드의 연관성을 공부함.
- 그 접근이 도움이 된 이유: 메서드가 캡슐화를 지키고 낮은 결합도 유지, 학습을 통해 이점을 알 수 있었음.
- 다음에도 반복하고 싶은 점: 코드가 직관적으로 깔끔하다고 느낄 때, 이게 실질적으로 어떤 아키텍처적 이득을 주는지 끝까지 원인을 분석하고 말로 설명하는 버릇을 유지하자.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- HttpResponse 클래스를 만들어서 필드와 getter를 정의했는데, 똑같은 기능을 Record를 이용해서 만들수 있다는 걸 알았다. record 사용했으면 품이 좀 덜 들어갔을 것 같다.
- 클래스를 getter, setter만 있게 만들거면 record를 사용해보자
- 아니면 클래스에 그 클래스를 다루는 메소드를 추가해보자. Response 파싱해서 넣는 메소드를 넣는 방식이라던가.
